# Qwen Voice Design Integration

**Date:** 2026-03-30
**Status:** Approved
**Scope:** Qwen TTS provider only — OpenAI and Gemini paths unchanged

---

## Problem

The current Qwen TTS flow assigns predefined system voices (Cherry, Ethan, etc.) to characters. These voices have fixed timbres, so characters of different ages and personalities sound similar. The `traits` field generated by the LLM is stored in the script JSON but never reaches the TTS API.

Qwen's `qwen3-tts-vd-2026-01-26` (Voice Design) model lets callers create a custom voice from a natural-language description, then use that voice for all synthesis. This is a better fit than the instruction-control approach because the character's personality is baked into the voice once at creation time, not approximated per-call.

---

## Goals

- Characters in Qwen-generated audiobooks sound distinctly different in age, personality, and emotion.
- The LLM writes a rich `voice_prompt` per character; no predefined voice pool is needed.
- Voice IDs are stored in the script JSON so resume and retry never recreate them.
- OpenAI and Gemini paths are completely unaffected.

---

## Non-Goals

- Voice preview / user approval before synthesis (automated pipeline only).
- Applying voice design to OpenAI or Gemini providers.
- Voice cloning from audio samples (`qwen3-tts-vc`).

---

## Architecture

### Data flow (Qwen provider)

```
VLM → LLM → Voice Design → TTS
```

Voice Design is a new stage inserted between LLM and TTS, only executed when `tts_provider == "qwen"`.

### Script JSON schema change (Qwen only)

| Field | OpenAI / Gemini | Qwen (new) |
|---|---|---|
| `voice` | `"alloy"` / `"Aoede"` | removed |
| `traits` | `"young girl, cheerful"` | removed |
| `voice_prompt` | — | LLM-generated description |
| `voice_id` | — | filled by Voice Design step |

Example `characters` entry for Qwen:
```json
{
  "name": "Bear",
  "voice_prompt": "A gentle elderly grandfather bear with a deep, low-pitched, slow-paced voice, warm and kind, soft-spoken, suitable for a children's picture book character.",
  "voice_id": null
}
```

`voice_id` is `null` after LLM stage and filled in during Voice Design stage. On resume, characters that already have a non-null `voice_id` are skipped.

## Components

### 1. LLM Service (`llm_service.py`)

**Change:** Qwen branch of `_system_prompt` generates `voice_prompt` instead of `voice` + `traits`.

- Remove `_VOICES["qwen"]` predefined pool.
- Tell the LLM to write `voice_prompt` as a natural-language sound description covering: age, gender, pitch, speed, emotion, characteristics, and role context.
- Guidelines embedded in the prompt (mirrors the Qwen instruction-control dimension table): be specific, multi-dimensional, objective, concise.
- Remove the 250-char utterance length rule for Qwen (no longer provider-specific — general rules apply).
- `voice_id` field is included in the schema with value `null`; the LLM always emits it as `null`.

Example prompt fragment:
```
"characters": [{"name": "...", "voice_prompt": "<natural language voice description>", "voice_id": null, "traits": "..."}]
- voice_prompt must describe: age, gender, pitch, speed, emotion, characteristics, and suitability.
  Example: "A cheerful 8-year-old girl with a bright high-pitched voice, fast-paced speech, excited and curious, sweet and childlike, suitable for children's animation."
- Every character must have a UNIQUE voice_prompt — describe each character distinctly.
```

### 2. New backend endpoint: `POST /tts/design-voices`

**Location:** `backend/routers/tts.py`

**Request body:**
```json
{
  "characters": [
    {"name": "Narrator", "voice_prompt": "...", "voice_id": null},
    {"name": "Bear",     "voice_prompt": "...", "voice_id": "bear_abc123"}
  ],
  "language": "zh",
  "qwen_api_key": "sk-..."
}
```

**Behaviour:**
- For each character where `voice_id` is `null`: call DashScope voice design API.
- Characters with an existing `voice_id` are passed through unchanged (idempotent on resume).
- Returns the full characters array with `voice_id` filled in.

**Response:**
```json
{
  "characters": [
    {"name": "Narrator", "voice_prompt": "...", "voice_id": "narrator_xyz789"},
    {"name": "Bear",     "voice_prompt": "...", "voice_id": "bear_abc123"}
  ]
}
```

### 3. TTS Service (`tts_service.py`)

**New function: `create_qwen_voice(client, character, language) -> str`**
- POST to `https://dashscope-intl.aliyuncs.com/api/v1/services/audio/tts/customization`
- Body: `model: "qwen-voice-design"`, `input.target_model: "qwen3-tts-vd-2026-01-26"`, `input.voice_prompt`, `input.preferred_name` (sanitised character name), `input.language`
- Returns `voice_id` string from `output.voice`.

**Change to `_call_qwen_segment`:**
- Model changes from `qwen3-tts-instruct-flash` to `qwen3-tts-vd-2026-01-26`.
- `language_type` field removed (not required by VD model).
- `_QWEN_VOICE_FALLBACK` map removed (no predefined voices; `voice_id` values are always valid).
- Everything else (throttling, WAV download, silence appending) unchanged.

**`generate_audio`:** No change — receives `voice_map: {name: voice_id}` as before; the values are now designed voice IDs instead of predefined names.

### 4. Script model (`lib/models/script.dart`)

- `Character` model: replace `voice` + `traits` with `voicePrompt` + `voiceId` for Qwen.
- `voiceFor(name)` returns `voiceId` for Qwen scripts, `voice` for others.
- Detect schema by presence of `voice_prompt` key.

### 5. Loading screen (`lib/screens/loading_screen.dart`)

Add Voice Design stage between LLM and TTS for `tts_provider == "qwen"`:

```
if (ttsProvider == 'qwen') {
  characters = await api.designVoices(
    characters: scriptMap['characters'],
    language: p.language,
  );
  scriptMap = {...scriptMap, 'characters': characters};
  // persist updated script to DB
}
```

Build `voice_map` from `voiceId` for Qwen, `voice` for others.

### 6. API Service (`lib/services/api_service.dart`)

New method `designVoices({required List characters, required String language})`:
- POST to `/tts/design-voices`
- Returns updated characters list.

## Error Handling

- Voice design failure for one character → log error, surface to user (same error screen as today). Do not partially proceed with some characters missing voices.
- TTS segment failure → existing error handling unchanged.
- On resume: `voice_id` already set → skip voice design for that character automatically.

## Migration

Existing Qwen audio versions (already generated, `status=ready`) are unaffected — they play from stored WAV files.

Existing Qwen audio versions with `status=error` or `status=generating` (stored script uses old `voice` field) will fail at Voice Design stage because they lack `voice_prompt`. The error screen will appear; the user must regenerate from scratch (re-run LLM stage).

No database migration required.
